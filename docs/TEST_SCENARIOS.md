# Сценарии ручной проверки

Этот документ нужен для ручной проверки поведения бота после изменений. Большую часть сценариев лучше проверять локально или на тестовом боте. На production не запускайте ручные обновления и рассылки без явной причины.

## Быстрые локальные проверки

```bash
python -m compileall .
python -m unittest discover -s tests -v
```

Эти проверки безопасны: они не обращаются к реальному Telegram API и не отправляют сообщения в чаты.

## Production safety

На сервере с systemd-сервисом не запускайте параллельно:

```bash
python main.py
```

Так появится второй polling-процесс с тем же `BOT_TOKEN`.

Безопасные команды проверки production:

```bash
systemctl status ait-bot.service --no-pager -l
journalctl -u ait-bot.service -n 120 --no-pager
ps -eo pid,user,cmd | grep -E '[/]opt/raspisanie_bot_ait/.*/python|[/]opt/raspisanie_bot_ait/main.py' | grep -v grep
```

На production команды `/update` и `/reparse` запускают реальную работу с расписанием. Используйте их только из админского аккаунта и только когда понятно, что рассылка допустима.

## Команды

### `/start`

- Админ в новом чате: чат должен зарегистрироваться.
- Не админ в новом чате: бот должен ответить, что регистрация ограничена.
- В уже зарегистрированном чате команда должна работать штатно.
- В незарегистрированном чате обычные команды чтения расписания должны быть заблокированы.

### `/today`

- В учебный день должен показываться корректный день и пары.
- В воскресенье бот должен сообщать, что пар нет.
- Если в тексте предмета есть символы `<`, `>` или `&`, сообщение не должно ломать HTML-разметку.

### `/tomorrow`

- В обычный день показывает следующий календарный учебный день.
- В субботу должен показывать понедельник, а не воскресенье.
- Если в базе есть текущая и следующая недели, должна выбираться нужная неделя по дате.

### `/update`

- Команда должна быть доступна только администраторам.
- При повторном вызове подряд должен сработать throttle.
- При наличии нового PDF в ответ должно приходить недельное сообщение.
- При отсутствии нового PDF бот должен вернуть, что нового расписания нет.
- На production перед проверкой убедитесь, что рассылка в зарегистрированные чаты допустима.

### `/reparse`

- Команда должна быть доступна только администраторам.
- Команда должна принудительно перепарсить текущий PDF даже при совпадающем хеше.
- После успешного перепарса бот должен прислать недельное расписание.
- При повторном вызове подряд должен сработать тот же throttle, что и у `/update`.
- На production эта команда может повторно отправить недельное расписание, поэтому не используйте её как обычную health-check команду.

## Контент расписания

### Окно в расписании

Если у группы отсутствует одна из пар, бот должен просто пропускать её номер без заполнения фиктивными значениями.

### Особые события

Проверь строки с ключевыми словами:

- `экзамен`
- `консультация`
- `дистант`
- `практика`
- `зачет`

Они должны нормально проходить через парсер и форматирование.

### Пустой день

Если для дня нет занятий, бот должен вернуть сообщение, что пар нет.

## Отказоустойчивость

### Telegram API недоступен с VPS

- `curl -4 -I https://api.telegram.org` с VPS зависает или падает с ошибкой соединения.
- Без обхода в логах появляются `TelegramNetworkError` или `Request timeout error`.
- После настройки `TELEGRAM_PROXY_URL` или `TELEGRAM_API_BASE_URL` polling должен стартовать.
- В логах ожидается строка `Run polling for bot ...`.
- Реальные значения прокси/gateway должны быть только в server-side `.env`, не в Git.

### Сайт недоступен

- В логах должна появляться ошибка запроса.
- Администратор должен получить уведомление о проблеме.

### Битый PDF

- Файл должен отбрасываться.
- Хеш последнего валидного файла не должен обновляться.
- Администратор должен получить уведомление.

### Слишком большой PDF

- PDF больше `MAX_PDF_SIZE` должен быть отклонён.
- Если сервер отдаёт большой `Content-Length`, скачивание не должно начинаться.
- Если размер превышен во время стриминга, временный файл должен удалиться.

### PDF со слишком большим числом страниц

- PDF с числом страниц выше `MAX_PDF_PAGES` должен быть отклонён.
- Текущие реальные PDF колледжа на 20+ страниц должны проходить, пока остаются в заданном лимите.

### Старый PDF без изменений

- Если хеш совпадает с предыдущим, бот не должен повторно делать рассылку.
- `/reparse` должен оставаться рабочим даже в этом случае.

## Безопасность

### Ссылки на PDF из HTML

Проверь сценарии:

- относительная ссылка на PDF с сайта колледжа;
- абсолютная HTTPS-ссылка на разрешённый host;
- абсолютная ссылка на внешний host;
- HTTP-ссылка вместо HTTPS.

Ожидание:

- разрешённые ссылки нормализуются;
- внешний host и HTTP отклоняются;
- редирект на внешний host отклоняется.

### Имя файла из `Content-Disposition`

Проверь сценарии:

- нормальное имя `schedule.pdf`;
- имя с обходом пути `../../evil.pdf`;
- файл с неверным расширением, например `schedule.txt`.

Ожидание:

- путь не выходит за пределы `downloads/`;
- не-`pdf` файл отклоняется.

### HTML в сообщениях

- Символы `<`, `>` и `&` в расписании должны отображаться как текст.
- Текст исключения в admin-уведомлении не должен превращаться в Telegram HTML-разметку.

## Что фиксировать после ручной проверки

Для каждого сбоя запишите:

- дату и время проверки;
- команду или сценарий;
- где проверяли: локально, тестовый бот или production;
- фрагмент лога без токенов и секретных URL;
- ожидание и фактический результат.
